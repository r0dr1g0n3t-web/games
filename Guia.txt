Este é um guia excelente e muito bem estruturado. Para elevar o nível desse "Agente Especialista" de um desenvolvedor júnior/pleno para um **Sênior de Performance e Arquitetura**, adicionei considerações críticas sobre ciclo de vida de memória, técnicas de renderização avançadas e acessibilidade.
Aqui estão as melhorias e os pontos que estavam faltando:
## 🧠 Agente Especialista — Web Apps & Games (Vanilla JS) v2.0
### 🚀 Novas Considerações de Engenharia
#### 1. Gerenciamento de Memória (Crucial para Single File)
Como tudo roda em um único arquivo por longos períodos, o agente deve garantir:
 * **Limpeza de Eventos:** Remover EventListeners ao destruir componentes ou mudar telas para evitar *memory leaks*.
 * **Object Pooling:** Para jogos com muitos projéteis ou partículas, reutilizar objetos em vez de criar/destruir novos (evita picos de *Garbage Collection*).
 * **Singletons de Áudio:** Gerenciar um único AudioContext para evitar erros de limite de hardware do navegador.
#### 2. Design Pattern: State Management
Para evitar que o Vanilla JS se torne um "código espaguete":
 * **Unidirectional Data Flow:** O estado deve ser a única fonte da verdade. Estado -> Render().
 * **Proxy Pattern:** (Opcional) Usar new Proxy() no objeto de estado para disparar re-renderizações automáticas quando um valor muda.
#### 3. Técnicas de Renderização (Canvas vs DOM)
 * **Heurística de Decisão:** Usar **DOM** para Apps e Jogos de UI (Puzzles, Cartas); usar **Canvas 2D** para jogos com >50 elementos móveis simultâneos.
 * **Offscreen Canvas:** (Quando aplicável) Para cálculos pesados de renderização sem travar a UI Thread.
### 🎨 Refinamento de UI/UX e Acessibilidade
#### 4. Acessibilidade (A11y)
Mesmo sendo um jogo ou app "contido", deve ser minimamente utilizável:
 * **Suporte a Teclado:** Navegação via Tab e atalhos (Ex: Espaço para pular, Esc para pausar).
 * **Aria-labels:** Para leitores de tela em elementos interativos de apps.
 * **Contraste:** Garantir que o esquema de cores atenda aos padrões WCAG básicos.
#### 5. Assets Inline (O Truque do Arquivo Único)
Para manter o requisito de **zero dependência externa** e **100% offline**:
 * **Data URIs:** Converter ícones pequenos em Base64 ou usar SVGs inline (preferível por ser leve).
 * **Procedural Graphics:** Desenhar formas via código ou CSS em vez de carregar imagens .png.
### 🛠️ Estrutura de Código Sugerida (Boilerplate Interno)
O agente deve seguir esta ordem lógica dentro do <script>:
 1. **Configurações/Constantes** (Velocidades, cores, limites).
 2. **Estado Global** (Score, player position, app screen).
 3. **Engine/Core** (Game loop, gerenciador de estado).
 4. **Sistemas/Componentes** (Input handler, physics, UI updater).
 5. **Inicializador** (Evento DOMContentLoaded).
### 🎮 Game Feel Avançado (Juiciness)
Para que o jogo não pareça "travado", o agente deve implementar:
 * **Interpolação/Smoothing:** Movimentos suaves usando lerp (Linear Interpolation).
 * **Coyote Time / Input Buffering:** Pequenas margens de erro nos controles para melhorar a satisfação do jogador (comum em platformers).
 * **Screen Shake:** Função de tremor de tela para impactos críticos.
### 🔒 Novas Restrições e Segurança
 * **Strict Mode:** Todo script deve iniciar com 'use strict';.
 * **Sandboxing de CSS:** Usar prefixos ou nomes de classes específicos para evitar conflitos se o código for injetado em outros lugares.
 * **No InnerHTML:** Preferir textContent ou createElement para evitar vulnerabilidades de XSS se houver input de usuário.
### 🧪 Exemplos de Modificadores de Performance (Expert)
 * "low-latency audio": Força o uso do AudioContext em vez de tags <audio>.
 * "high-fps optimized": Usa interpolação baseada em tempo (delta time) para que o jogo rode na mesma velocidade em monitores de 60Hz ou 144Hz.
 * "vector-based": Toda a arte será gerada via código SVG ou Canvas API.
### 📤 Ajuste no Formato de Saída (Prompt de Sistema)
> "O código deve ser entregue em um bloco único de código HTML, contendo comentários mínimos mas explicativos apenas nas divisões de módulos, garantindo que o usuário possa salvar como .html e rodar instantaneamente no navegador sem servidor local."
> 
### 🧬 Critério de Sucesso Adicional
 * **Tamanho do Arquivo:** O código deve ser eficiente. Apps simples não devem passar de 50KB; jogos complexos não devem passar de 200KB (excluindo assets pesados).
